# Methodology References — util-metamodel-migration

Internal reference. Not copied to projects. Explains the rationale behind the skill's design decisions.

---

## Why tiered detection with confidence scoring?

Documentation folders are organic — they grow without naming conventions and accumulate synonyms. A file named `process-guide.md` might be a business process doc, a developer guide, or a pipeline step guide. A single-signal detector would produce too many false positives.

The three-tier system (filename → folder → content signals) borrows from information retrieval precision/recall principles:
- Tier 1 (filename) has high precision but misses files with non-standard names
- Tier 2 (folder) is a contextual signal — the surrounding structure adds evidence
- Tier 3 (content signals) is the disambiguator — the actual headings prove the type

Requiring ≥2 tiers to agree before emitting a fix block reduces false positive rate significantly. Single-tier matches are surfaced as Low confidence so the user can verify — the skill surfaces the signal without asserting certainty.

---

## Why read only the first 50 lines for content signals?

Large repos can have hundreds of markdown files. Reading every file in full would make the scan impractically slow. The first 50 lines of a well-structured markdown document contain the H1 title, the intro paragraph, and the first 2-3 section headings — which is enough to identify the artefact type with high confidence.

The canonical skill pattern (from the kit) puts the most identifying headings early in every document. This makes first-50-line detection reliable for docs generated by the kit's skills.

---

## Why atomic fix blocks instead of a simple table?

A migration is only safe if it's atomic: move the file AND repair all inbound links in one operation. If a user applies the `git mv` but misses a `sed` repair, the repo ends up with broken links — worse than before the migration.

The atomic fix block design (one copy-paste per file move) ensures that every migration operation is complete and self-consistent. The user can choose which blocks to apply (they may not want to migrate everything at once) but each block they do apply leaves the repo in a consistent state.

This pattern is analogous to database migrations: each migration is a transaction — apply fully or not at all.

---

## Why report-only, not auto-fix?

The skill has no way to know whether a detected "misplacement" is intentional. A team might have `docs/runbooks/` instead of `docs/ops/runbooks/` because they have a different convention or because a specific tool expects that path. Auto-fixing would break their setup.

Report-only with atomic blocks gives the team full control: they review each finding, decide whether to apply it, and run the commands themselves. This is the same philosophy as `util-metamodel-audit`.

---

## Why DDD candidates don't get git mv commands?

The domain layer (`docs/domain/`) requires specific document structure (BC-NN IDs, GT-NN IDs, aggregate definitions, etc.) that the domain skills scaffold. Moving a raw `glossary.md` to `docs/domain/glossary/glossary.md` without running `domain-glossary` would place an unstructured file at the canonical path — which would then FAIL the util-metamodel-audit mandatory-sections check.

The correct migration sequence for DDD candidates:
1. Run the domain skill in scaffold + seed mode (it reads the existing file and imports its content)
2. The domain skill creates the properly structured file at the canonical path
3. The old file can then be archived or removed

This is why §4 proposes "run this skill" rather than a `git mv`.
